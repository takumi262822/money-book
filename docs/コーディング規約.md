# コーディング規約

> プロジェクト: **kakeibo-app（家計簿アプリ）**  
> 作成者: Takumi Harada  
> 作成日: 2026/04/06  

---

## 1. 概要

本ドキュメントは kakeibo-app における JavaScript のコーディング規約を定めます。  
全ファイルはこの規約に沿って実装・レビューすることを原則とします。

---

## 2. フォルダ構成

```
src/
├── main.js                     # エントリーポイント
├── constants/
│   ├── app-constants.js        # アプリ全体の定数（カテゴリ・ストレージキー等）
│   └── code-definitions.js     # コード定義（MODAL_STATE・ERROR_CODES 等）
├── core/
│   ├── kakeibo-engine.js       # アプリ全体の進行制御・イベント委譲
│   └── budget-service.js       # 予算の CRUD ビジネスロジック
├── managers/
│   ├── chart-manager.js        # Chart.js ラッパー（グラフ描画）
│   ├── csv-manager.js          # CSV エクスポート処理
│   ├── state-manager.js        # UI 状態管理（選択年月・モーダル状態）
│   └── storage-manager.js      # localStorage の読み書きラッパー
├── ui/
│   ├── components.js           # 再利用可能な DOM 生成関数
│   ├── header.js               # ヘッダー HTML 生成
│   └── footer.js               # フッター HTML 生成
└── utils/
    ├── formatter.js            # 金額・日付などのフォーマット関数
    ├── validator.js            # 入力バリデーション関数
    └── xss.js                  # XSS サニタイズ関数
```

### 各ディレクトリの責務

| ディレクトリ | 責務 |
|---|---|
| `constants/` | 変更しない定数・コード定義のみを置く。ロジックを含めない |
| `core/` | ビジネスロジックの核。DOM 操作は最小限に抑える |
| `managers/` | 単一の外部機能・状態に特化したクラス |
| `ui/` | DOM 生成・描画のみ。ロジックを含めない |
| `utils/` | プロジェクト横断で使える純粋関数 |

## 3. 命名規則

### 3-1. ファイル名

- **kebab-case** を使用する

```
kakeibo-engine.js  ✅
kakeiboEngine.js   ❌
```

### 3-2. クラス名

- **PascalCase** を使用する

```js
export class KakeiboEngine { }   ✅
export class kakeiboEngine { }   ❌
```

### 3-3. 関数・メソッド・変数名

- **camelCase** を使用する

```js
function validateRecord() { }   ✅
function validate_record() { }  ❌
```

### 3-4. 定数名

- **UPPER_SNAKE_CASE** を使用する
- モジュールスコープの定数（`const`）はすべて `constants/` に集約する

```js
export const STORAGE_KEYS = { RECORDS: 'moneybook_records' };  ✅
const storageKeys = { records: 'moneybook_records' };           ❌
```

### 3-5. プライベートメソッド

- クラス内の内部専用メソッドは `_` プレフィックスを付ける

```js
_renderShell() { }   ✅  （外部から呼ばない）
renderShell() { }    ✅  （外部から呼ぶ）
```

## 5. コメント規約

### 5-1. ファイルヘッダー

全ファイルの先頭に JSDoc ブロックコメントを記述する。

```js
/**
 * <ファイルの役割を一行で>
 * @author Takumi Harada
 * @date YYYY/MM/DD
 */
```

### 5-2. メソッド区切りコメント

メソッドの手前にメソッドの目的を示すセクションコメントを付ける。

```js
// アプリを初期化して DOM にレンダリングする
init() {
  ...
}
```

### 5-3. if 文コメント

**すべての `if` 文の直前行**に、何を判定しているかを日本語で説明するコメントを付ける。

```js
// レコードが0件の場合は空状態メッセージを表示して早期リターンする
if (records.length === 0) {
  this._showEmpty();
  return;
}
```

### 5-4. 複雑なロジックのコメント

自明でないロジック（計算式・正規表現・タイマー等）には理由を補足する。

```js
// ChartManager がシールドゼロを検知した場合にゲームオーバーフラグを立てる
// Fisher-Yates に近い簡易シャッフルで QUESTION_COUNT 問を抽出
```

## 6. モジュール設計

### 6-1. ES Modules 使用

`import` / `export` を使用する。`require()` は使用しない。

```js
import { StorageManager } from '../managers/storage-manager.js';
export class KakeiboEngine { }
```

### 6-2. 名前付きエクスポートを原則とする

```js
export class KakeiboEngine { }          ✅
export default class KakeiboEngine { }  ❌（デフォルトエクスポートは避ける）
```

### 6-3. import パス

拡張子 `.js` を明示する。

```js
import { StorageManager } from '../managers/storage-manager.js';  ✅
import { StorageManager } from '../managers/storage-manager';     ❌
```

## 7. クラス設計

### 7-1. 単一責任の原則

1 クラス = 1 つの責務。複数の役割を持たせない。

| クラス | 責務 |
|---|---|
| `KakeiboEngine` | イベント委譲・画面更新の指揮 |
| `BudgetService` | 予算データの CRUD |
| `StorageManager` | localStorage の読み書きのみ |
| `ChartManager` | Chart.js の描画のみ |

### 7-2. コンストラクタ

依存クラスのインスタンス化と DOM 参照の取得のみを行う。ロジックは書かない。

```js
constructor() {
  this.storage = new StorageManager();
  this.state   = new StateManager();
}
```

## 8. セキュリティ

### 8-1. XSS 対策

ユーザー入力を DOM に反映する際は **必ず** `escapeHtml()` を通す。  
`innerHTML` へのユーザー入力直接代入は禁止。

```js
import { escapeHtml } from '../utils/xss.js';

// ✅ 正しい使用
element.textContent = escapeHtml(userInput);

// ❌ 禁止
element.innerHTML = userInput;
```

### 8-2. localStorage の入出力

- 書き込み前にバリデーションを通す
- 読み込み後は `JSON.parse` を `try/catch` で包む

```js
try {
  const data = JSON.parse(localStorage.getItem(key));
} catch (e) {
  console.warn('読込失敗:', e.message);
  return [];
}
```

## 9. バリデーション

### 9-1. バリデーション関数の場所

入力バリデーションは `src/utils/validator.js` に集約する。

### 9-2. 早期リターンパターン

バリデーション失敗時は処理の先頭で早期リターンする。

```js
// バリデーションエラーが1件以上あれば例外を投げる
if (errors.length > 0) {
  throw new Error(errors.join(', '));
}
```

### 9-3. エラーコード

エラーメッセージは文字列リテラルでなく、`ERROR_CODES` 定数を使う。

```js
errors.push(ERROR_CODES.INVALID_DATE);   ✅
errors.push('日付が不正です');            ❌
```

## 10. Git 運用

### 10-1. コミットメッセージ

プレフィックスを付けて種別を明示する。

| プレフィックス | 用途 |
|---|---|
| `feat:` | 新機能追加 |
| `fix:` | バグ修正 |
| `refactor:` | リファクタリング（動作変更なし） |
| `docs:` | ドキュメント更新 |
| `test:` | テスト追加・修正 |
| `chore:` | 設定・ツール類の変更 |

```
feat: 予算設定モーダルを追加
fix: CSV エクスポート時の文字化けを修正
refactor: ifブロックに判定内容コメントを追加
docs: コーディング規約を追加
```

### 10-2. ブランチ

| ブランチ | 用途 |
|---|---|
| `master` | 本番リリース用 |
| `feature/xxx` | 機能開発 |
| `fix/xxx` | バグ修正 |
